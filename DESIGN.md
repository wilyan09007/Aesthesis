## 1. Vision

**Aesthesis is a neural UX read on any product experience: drop a screen recording, get a brain-grounded analysis.**

> Note (DESIGN.md ¬ß17, 2026-04-25): Aesthesis used to be an A/B comparison product. The pivot to single-video collapsed every "two videos in, winner out" path. The vision below has been rewritten; the pipeline-level sections that follow describe the new shape. The TRIBE service (¬ß5.x) is unchanged ‚Äî it's per-video by construction.

Give it a deployment URL or upload an MP4. The pipeline reads the demo through Meta's TRIBE v2 brain encoding model ‚Äî a multimodal foundation model trained on 451+ hours of fMRI data from 720+ humans. TRIBE v2 returns a per-second timeline of predicted neural activity across six interpretable cognitive/emotional regions. We translate that timeline into actionable UX insights timestamped to specific moments in the demo, then summarise the brain's overall reaction.

The product answers the question: *"What did the human brain notice in this demo, and exactly when did attention, friction, or intent fire?"*

Pitch line: "Demo anything. See the analysis."

---

## 2. Why Now

Three things just became true at once.

1. **TRIBE v2 was open-sourced March 2026.** Meta released the first foundation model that predicts whole-brain fMRI responses to naturalistic video, audio, and text ‚Äî won Algonauts 2025, 70√ó resolution jump over prior work, ~70k cortical vertices on the fsaverage5 mesh. Non-commercial license, weights on Hugging Face. Until now, "AI predicts the brain" was a research toy. Now it's a deployable model with documented inference.
2. **Browser agents finally work.** BrowserUse, Playwright + LLM, Anthropic Computer Use ‚Äî autonomous agents can drive a real browser through a real product flow with usable reliability.
3. **Vision-language models can narrate UX.** GPT-4V, Claude with vision, Gemini Pro Vision ‚Äî they can watch a screen recording and describe what a user sees and likely feels, frame by frame.

Stacking all three gives you something that didn't exist 12 months ago: an automatable neural UX read that runs in seconds, not weeks.

The hackathon question: can we make a demo that lands the "whoa" before the cynicism kicks in?

### 3.5 TRIBE v2 video input pathway

The screen recording from each browser agent feeds directly into TRIBE v2 as an MP4 file. No vision-language narrator, no intermediate text ‚Äî the underlying `tribev2` package supports `model.get_events_dataframe(video_path=...)` natively. The deployment needs a new `/process_video_timeline` endpoint to expose this path (¬ß5.15); the underlying capability is already there.

**Why direct video (vs. earlier narrator-bridge plan):**

- **No hallucination layer.** The model sees the actual pixels, not a VLM's interpretation of them.
- **In-distribution.** TRIBE was trained on naturalistic film, audio, and text. Video is exactly what it was built to consume. Most of the ¬ß5.6 text-pathway corrections become obsolete (they were patches for running TRIBE on out-of-distribution text). VIFS + PINES become valid signals again.
- **Stronger demo pitch.** *"The brain reacts directly to the screen recording"* beats *"AI describes the screen, brain reacts to the description."*
- **Less moving parts.** One fewer LLM call, one fewer place for things to go wrong.

**Tradeoffs (acknowledged, not blockers):**

- **Retinotopy blur.** V-JEPA 2 spatially averages patch tokens before fusion. UI-level precise position info (cursor, text placement) lives in retinotopic V1/V2/V4 and gets blurred. **Mostly fine for our keyset** ‚Äî 7 of the 8 UX-tuned ROIs (`aesthetic_appeal`, `cognitive_load`, `trust_affinity`, `reward_anticipation`, `motor_readiness`, `surprise_novelty`, `friction_anxiety` ‚Äî see ¬ß5.16) are higher-level integration regions that don't depend on pixel-level position info. The one exception is `visual_fluency`, which uses `_Vis_` (Yeo Visual = retinotopic). Its signal will be magnitude-attenuated by the blur. Acceptable ‚Äî fluency direction (high/low) survives even if exact magnitude is muddied.
- **Wall time.** ~5‚Äì15s for a 60s 720p clip vs. ~2‚Äì5s for text. Acceptable ‚Äî runs in the background while the live frame stream gives the user something to watch.
- **VRAM.** Heavier ‚Äî verify on existing GPU during the Phase 0 spike (¬ß5.15.6).

We deploy our own TRIBE v2 service for this ‚Äî see ¬ß5 for the reference architecture (patterns to follow), ¬ß5.15 for the video pathway specifics. We're not consuming any third-party deployment.

**Phase 0 spike (1‚Äì2 hours) verifies that the `tribev2` package's video pathway works on UI screen recordings, that we get sensible ROI signal back, and that VRAM fits on the GPU we have available.** Pass ‚Üí ship. Fail ‚Üí escalate; this is the only path designed.

## 4. Component Specs

### 4.0 Step structure (D10)

Two stages, with Step 1 fully skippable:

| | **STEP 1 ‚Äî Capture** (optional) | **STEP 2 ‚Äî Assess** (always) |
|---|---|---|
| Inputs | URL A, URL B, Goal | `demo_a.mp4`, `demo_b.mp4`, Goal (optional) |
| Components | ¬ß4.1 Browser Agent, ¬ß4.2 Screen Recorder, ¬ß4.2b Live Streamer | ¬ß4.4 TRIBE service, ¬ß4.5 Insight Synthesizer |
| Outputs | Two MP4s + action logs | Brain timelines + insights + verdict |
| Wall time | ~30s (parallel agent runs) | ~15‚Äì25s (serial TRIBE + Gemini) |
| Skip mode | ‚Äî | ¬ß4.7 MP4 Upload component lets users drop two MP4s directly, bypassing Step 1 |
| UI | ¬ß4.6 wizard panel: capture mode | ¬ß4.6 wizard panel: assess mode ‚Üí results mode |

The Goal field carries through both modes:
- **Capture mode:** Goal is required, parameterizes the BrowserUse system prompt (¬ß4.1).
- **Skip mode:** Goal is optional. Without it, Gemini synthesizes against a generic "first-time visitor" frame. With it, Gemini focuses insights on the user's stated concern.

### 4.1 Browser Agent ‚Äî demo runner (Step 1, optional)

**Job:** Given a URL and a flow goal, drive a real browser through a meaningful demo of the product.

**Stack candidates:**

| Option | Pro | Con |
|---|---|---|
| **BrowserUse** (LLM-driven Playwright) | Autonomous, picks its own actions, "watch the AI demo your product" is a great hackathon visual | Slower, less predictable, can get lost on novel UIs |
| **Playwright + scripted flow** | Deterministic, fast, never breaks | Each URL needs a hand-written script, kills the "any URL" pitch |
| **Anthropic Computer Use** | Most general, can handle weird UIs | Heavy, expensive per run, overkill for browser |
| **Hybrid: agent generates plan, scripted execution** | Best reliability/coverage tradeoff | More code |

**Recommended for hackathon:** BrowserUse with a tight, **user-targeted system prompt** (D8). Instead of open exploration, the user provides a `goal` string ("evaluate signup flow," "find pricing and compare tiers," "use the demo"). BrowserUse's prompt is parameterized on this goal. Targeted agents are dramatically more reliable than open exploratory ones.

**Hard timeout: 30 seconds wall-clock per agent** (D7). Targeted goals usually finish faster than open exploration; cap keeps clips short for TRIBE inference cost.

> **Recording quality matters.** TRIBE sees the actual pixels via V-JEPA 2 ‚Äî make sure the recorder produces clean 720p+ footage with no artifacting (see ¬ß4.2). Cursor visibility matters too; enable a cursor highlight extension before recording so the agent's clicks are legible to both human viewers and the model.

**System prompt template** (parameterized on user-provided `goal`):
```
You are a first-time visitor to {url}. Your task: {goal}.

Constraints:
- You have 30 seconds wall-clock total. Move efficiently.
- Behave like a curious-but-impatient real human. Read the hero, click
  the obvious CTA, fill 1-2 fields if a form appears.
- Do NOT refresh, open dev tools, type sensitive info, or open new tabs.
- If you complete the goal early, that's fine ‚Äî finish and stop.
- If you hit an unexpected page (404, login wall), stop immediately.
```

**Kill mechanism (D1 ‚Äî load-bearing).** BrowserUse's run loop is documented as not respecting internal timeouts (issues #1157, #3615, etc.). Wrap BrowserUse in a subprocess; parent enforces timeout via `asyncio.wait_for` on the subprocess; on timeout, **SIGKILL** the subprocess. Then verify Chromium child processes are dead too (Playwright leaves zombies sometimes), kill them if alive. Critical: see SIGKILL regression test in ¬ß15 R4.

```python
# pseudocode
proc = await asyncio.create_subprocess_exec(
    "python", "-m", "aesthesis.browser_agent", "--url", url, "--goal", goal,
    stdout=PIPE, stderr=PIPE,
)
try:
    await asyncio.wait_for(proc.wait(), timeout=90)  # 30s agent + 60s buffer
except asyncio.TimeoutError:
    proc.kill()                          # SIGKILL the parent
    await proc.wait()
    _kill_chromium_zombies(proc.pid)    # then any orphan Chromium
    return load_pre_recorded_fallback(url)
```

**Output:** `demo_<a|b>.mp4` (1080p, 30fps, ‚â§30s) and `actions_<a|b>.jsonl` (every click/scroll/input with timestamps).

### 4.2 Screen Recorder (MP4) ‚Äî Step 1, optional

**Job:** Capture the browser tab as a high-quality MP4 file while the agent drives it. This MP4 is the artifact that gets POSTed to TRIBE v2. Separate from ¬ß4.2b's live frame stream, which is purely for frontend progress.

**Options:**
- **Playwright video recording** (built-in, just enable `recordVideo` in context) ‚Äî easiest, MP4 out, syncs with actions.
- **Puppeteer screencast** + ffmpeg ‚Äî works but more glue.
- **CDP (Chrome DevTools Protocol) Page.startScreencast** + ffmpeg encode ‚Äî frame-by-frame control, more code.

**Recommended:** Playwright `recordVideo`. Already integrated if BrowserUse uses Playwright underneath.

**Output spec:** 1280√ó720 minimum (1920√ó1080 if GPU budget allows), 30fps, **‚â§30 seconds per demo (D7)**. The MP4 quality matters because TRIBE's V-JEPA 2 encoder consumes these frames directly ‚Äî artifacting hurts the brain prediction. 30s √ó 30fps = ~900 frames, manageable for V-JEPA 2's 64-frame-per-4s window without truncation issues.

### 4.2b Live Frame Streamer (adaptive, 2 fps floor) ‚Äî Step 1, optional

**Job:** While the browser agent runs, push a live image stream of the tab to the frontend so the user sees the agent working in real time. Separate artifact from the ¬ß4.2 MP4 (which goes to TRIBE) ‚Äî this is purely a UX channel.

**Why:** Text progress ("Agent A on pricing page...") is dead UI. Watching the agent navigate live is itself half the demo magic. Two side-by-side live streams during the run, then the same panels swap to high-quality MP4 players + brain charts when results are ready.

**Hard requirement (D9): never dip below 2 fps.** Prior bad-WiFi runs have failed this. The streamer is **adaptive** ‚Äî it monitors actual delivered FPS and steps quality down (resolution + JPEG quality + frame rate) before the floor is breached.

**Quality tier ladder** (5 tiers, T0 ‚Üí T4):

| Tier | Resolution | JPEG quality | `everyNthFrame` | Target FPS | Frame size (~) |
|---|---|---|---|---|---|
| T0 (default) | 800 √ó 600 | 60 | 3 | 10 | 50 KB |
| T1 | 640 √ó 480 | 50 | 3 | 10 | 32 KB |
| T2 | 480 √ó 360 | 40 | 5 | 6 | 18 KB |
| T3 | 320 √ó 240 | 35 | 10 | 3 | 9 KB |
| **T4 (floor)** | 240 √ó 180 | 25 | 15 | **2** | 4 KB |

CDP base rate is 30 fps; `everyNthFrame=N` means "send every Nth frame" ‚Üí effective rate = 30 / N.

**Adaptation rules:**

- Backend tracks `Page.screencastFrame` event timestamps; computes actual delivered FPS over a rolling 2s window (per stream)
- Every 2s: if actual FPS < `target √ó 0.8` ‚Üí **step down one tier** (call `Page.stopScreencast` then `Page.startScreencast` with new params; brief ~100ms gap)
- Every 4s of sustained `actual FPS > target √ó 1.2` AND not at T0: **step up one tier** (hysteresis prevents oscillation)
- At T4 (the floor): if FPS *still* drops below 2 ‚Üí emit `{type: "stream_degraded"}` event to frontend ‚Üí UI shows "live stream degraded (network)" overlay but never disconnects. MP4 still records server-side regardless.

**Backend sketch:**

```python
TIERS = [
    {"width": 800, "height": 600, "quality": 60, "everyNthFrame": 3,  "target_fps": 10},
    {"width": 640, "height": 480, "quality": 50, "everyNthFrame": 3,  "target_fps": 10},
    {"width": 480, "height": 360, "quality": 40, "everyNthFrame": 5,  "target_fps": 6},
    {"width": 320, "height": 240, "quality": 35, "everyNthFrame": 10, "target_fps": 3},
    {"width": 240, "height": 180, "quality": 25, "everyNthFrame": 15, "target_fps": 2},
]

class AdaptiveStreamer:
    def __init__(self, cdp, ws, version_id):
        self.cdp = cdp; self.ws = ws; self.version = version_id
        self.tier = 0
        self.frame_times = collections.deque(maxlen=20)
        self.good_streak_started_at = None

    async def start(self):
        await self._apply_tier()
        self.cdp.on("Page.screencastFrame", self._on_frame)
        asyncio.create_task(self._adapt_loop())

    async def _apply_tier(self):
        try: await self.cdp.send("Page.stopScreencast")
        except: pass
        p = TIERS[self.tier]
        await self.cdp.send("Page.startScreencast", {
            "format": "jpeg", "quality": p["quality"],
            "maxWidth": p["width"], "maxHeight": p["height"],
            "everyNthFrame": p["everyNthFrame"],
        })

    async def _on_frame(self, params):
        now = time.monotonic()
        self.frame_times.append(now)
        await self.ws.send_json({
            "type": "frame", "version": self.version,
            "frame_b64": params["data"], "ts_ms": now * 1000, "tier": self.tier,
        })
        await self.cdp.send("Page.screencastFrameAck",
                            {"sessionId": params["sessionId"]})

    async def _adapt_loop(self):
        while True:
            await asyncio.sleep(2.0)
            actual = self._measure_fps()
            target = TIERS[self.tier]["target_fps"]
            if actual < target * 0.8 and self.tier < len(TIERS) - 1:
                self.tier += 1; await self._apply_tier()
                self.good_streak_started_at = None
                if self.tier == len(TIERS) - 1 and actual < 2:
                    await self.ws.send_json({"type": "stream_degraded",
                                             "version": self.version})
            elif actual > target * 1.2 and self.tier > 0:
                if self.good_streak_started_at is None:
                    self.good_streak_started_at = time.monotonic()
                elif time.monotonic() - self.good_streak_started_at > 4.0:
                    self.tier -= 1; await self._apply_tier()
                    self.good_streak_started_at = None
            else:
                self.good_streak_started_at = None

    def _measure_fps(self):
        if len(self.frame_times) < 2: return 0
        window = self.frame_times[-1] - self.frame_times[0]
        return len(self.frame_times) / window if window > 0 else 0
```

**Frontend** renders frames in `<img>` element via base64 data URL. Listens for `stream_degraded` to show overlay (just CSS). Shows current tier as a small indicator in dev mode.

### 4.3 V-JEPA 2 Direct Video Input ‚Üí TRIBE

**Job:** Feed each screen recording (`demo_<a|b>.mp4`) directly to TRIBE v2's native video pathway. No VLM, no intermediate text ‚Äî pure video ‚Üí brain.

**How:** The `tribev2` Python package supports `model.get_events_dataframe(video_path=...)` which automatically:

1. Demuxes audio from the MP4
2. Runs WhisperX for word-level transcript + alignment
3. Encodes video frames through DINOv2 + V-JEPA 2-Giant (2 Hz, D=1280, 64 frames per 4s window)
4. Encodes audio through Wav2Vec-BERT
5. Encodes the WhisperX transcript through LLaMA 3.2
6. Aligns all three streams at 2 Hz, fuses through Transformer, predicts fMRI

A video input automatically activates **all three modalities**, not just video. Silent video ‚Üí audio + transcript streams are effectively empty but still concatenated.

We deploy our own TRIBE v2 service that exposes a `/process_video_timeline` endpoint wrapping `model.predict(events=df)` and the ROI aggregation. ¬ß5 is the reference architecture (FastAPI + Redis + ARQ + GPU worker, modeled on the YNeurotrading patterns); ¬ß5.15 is the video pathway implementation specifics.

**Tradeoffs (acknowledged, not blockers):**

- **Retinotopy blur:** V-JEPA 2 spatially averages patch tokens before fusion. UI-precise position info (cursor, text placement) lives in retinotopic V1/V2/V4 and gets blurred. **Affects 1 of our 8 ROIs** ‚Äî `visual_fluency` (uses `_Vis_`) is partially retinotopic and degrades in magnitude. The other 7 (`aesthetic_appeal`, `cognitive_load`, `trust_affinity`, `reward_anticipation`, `motor_readiness`, `surprise_novelty`, `friction_anxiety` ‚Äî see ¬ß5.16) are higher-level integration regions, unaffected. We're asking "did the user feel uncertainty / engagement / trust" not "where did the user look."
- **Wall time:** ~5‚Äì15s for a 60s 720p clip vs. ~2‚Äì5s for text. Acceptable ‚Äî runs in the background while the ¬ß4.2b live frame stream gives the user something to watch in the meantime.
- **VRAM:** V-JEPA 2-Giant + DINOv2 + Wav2Vec-BERT + LLaMA 3.2 + TRIBE simultaneously is heavier than text-only. Verify on existing GPU during Phase 0 spike (¬ß5.15.6).
- **Service work:** new `/process_video_timeline` endpoint required (¬ß5.15).
- **Hemodynamic lag:** TRIBE pre-applies a 5s "into the past" offset to compensate for BOLD delay. Affects how brain frames align to UI moments in visualizations ‚Äî see open question ¬ß10.10.

**Output:** `brain_<a|b>.json`, identical schema to `/process_timeline` response. Same data structure the Insight Synthesizer (¬ß4.5) expects.

**Phase 0 spike (1‚Äì2 hours):** before depending on this, verify package surface, monkey-patch coexistence, in-distribution behavior on UI recordings, and VRAM headroom (¬ß5.15.6). Pass ‚Üí ship. Fail ‚Üí escalate; this is the only bridge designed.

### 4.5 Insight Synthesizer (worked example)

**Job:** Convert the brain timeline + screenshots + agent action log into 3‚Äì5 timestamped, actionable insights per version, plus a head-to-head verdict.

**The hard part of the entire system lives here.** "Voxel 14,237 lit up at t=4.2s" is not insight. The translation layer turns brain signals into UX language. Below is the full pipeline with a worked example so the mechanism is unambiguous.

#### What we read from TRIBE

For a 60-second demo video, the deployment returns a per-1.5s matrix. Rows = TRs (~40 for 60s). Columns = the 8 UX ROIs (¬ß5.16). Each cell is a normalized activation:

```
            appeal  fluency  load  trust  reward  motor  surprise  friction
t=0.0s       0.30     0.50   0.20   0.40    0.30   0.00     0.10      0.10
t=1.5s       0.40     0.45   0.30   0.40    0.50   0.10     0.20      0.30
t=3.0s       0.55     0.40   0.40   0.40    0.80   0.20     0.30      0.20
t=4.5s       0.60     0.40   0.50   0.45    0.70   0.30     0.20      0.20
...
t=18.0s      0.20     0.30   1.20   0.30    0.20   0.10     0.90      1.40   ‚Üê friction spike
t=19.5s      0.25     0.30   1.10   0.30    0.20   0.10     0.80      1.20
...
```

Plus per-1.5s **composites** (`appeal_index`, `conversion_intent`, `fluency_score`, `trust_index`, `engagement_depth`, `surprise_polarity`, `memorability_proxy`, `ux_dominance` ‚Äî ¬ß5.16.3) and per-6s **window composites** (`flow_state`, `decision_clarity`, `bounce_risk`, `hook_strength`, `aesthetic_dwell`, `friction_burst` ‚Äî ¬ß5.16.4) and pairwise connectivities from `PAIRS_UX` (¬ß5.16.5). Full schema in ¬ß5.3.

This is the raw signal. Interpretable but not yet actionable.

#### Step 1 ‚Äî Feature extraction (deterministic, Python ‚Äî no AI here)

Walk the matrix and pull out **events worth talking about**. Hand-coded rules:

| Event type | Rule | Example |
|---|---|---|
| **Spike** | one ROI jumps > 1.5œÉ above its own delta noise floor | "friction_anxiety spiked +1.1 at t=18s" |
| **Dominant shift** | the ROI with the highest activation changed | "dominant flipped reward_anticipation ‚Üí friction_anxiety at t=18s" |
| **Sustained window** | same ROI dominant for 3+ TRs in a row | "cognitive_load dominant t=14s‚Äìt=20s" |
| **Co-movement** | two ROIs both rise (or both fall) in the same TR | "friction_anxiety + cognitive_load rose together at t=18s" |
| **Trough moment** | `appeal_index` < -0.3 AND `friction_anxiety` > 0.7 | "anxiety peak at t=22s" |
| **Window: flow** | `flow_state` window composite triggers | "sustained absorbed engagement at t=30s‚Äì36s" |
| **Window: bounce risk** | `bounce_risk` window composite triggers | "confused user about to leave at t=14s‚Äì20s" |

Each event becomes a structured row:

```json
{
  "version": "A",
  "timestamp_s": 18.0,
  "type": "spike",
  "roi": "friction_anxiety",
  "magnitude": 1.4,
  "co_events": ["dominant_shift_reward_anticipation_to_friction_anxiety", "spike_cognitive_load"],
  "screenshot_path": "demo_a_t18.jpg",
  "agent_action_at_t": "clicked Sign Up button"
}
```

Typical 8‚Äì15 events per 60s video. That's the menu the LLM picks from.

#### Step 2 ‚Äî LLM turns events into insights

We feed the LLM **three things per event:**

1. The structured event record (above)
2. The screenshot at that timestamp (extracted from MP4 via ffmpeg)
3. The agent action log around that timestamp ("scrolled, clicked X, paused 2s")

**Model: Gemini.** Gemini 2.0 Flash for dev iteration (cheap, fast, supports vision). Gemini 2.5 Pro for final demo polish if Flash signal-to-noise isn't quite there. Per-comparison cost on Flash: ~$0.002‚Äì0.004. Order of magnitude cheaper than GPT-4o or Claude Opus, which matters for iterating prompts + thresholds during the build.

**Concrete prompt + JSON schemas (R1).** Vague prompts produce vague insights. Precise input/output contracts produce demoable insights.

**Prompt** (use this verbatim, iterate from here):

```
You are analyzing how a first-time visitor's brain responded to a website
demo. The user's stated goal was: "{user_goal}".

You will see {N} brain events from a 30-second screen recording. Each event
includes the brain signal that fired, the screenshot at that moment, and
what the user (an AI agent acting as a real user) did just before.

Your job: for each event, produce ONE insight in this format:
- ux_observation: one sentence explaining what on the screen likely caused
  this brain response. Be concrete ‚Äî name the UI element, the copy, the
  layout choice.
- recommendation: one sentence with a specific, implementable change.
- cited_brain_features: list of the brain feature keys you used.
- cited_screen_moment: one phrase describing the visible UI moment.

Hard constraints:
- One insight per event. Do not aggregate or summarize across events.
- Each insight MUST cite at least one brain feature key.
- Each recommendation MUST be concrete enough that an engineer could
  implement it without asking questions. "Improve UX" is not a recommendation;
  "move the trial CTA above the fold and increase its weight to match the
  paid tiers" is.
- Do NOT use product-marketing language. No "delightful," "intuitive,"
  "seamless," "frictionless," "leverage," "robust."
- Do NOT speculate beyond what the brain features support. If fear spiked
  but you can't tell from the screenshot why, say "screenshot does not
  reveal cause; possible candidates: X, Y."
- Output valid JSON conforming exactly to the output schema. No prose
  outside the JSON.
```

- Cite at least 2 specific timestamped moments in the summary

**Example output:**

#### Step 3 ‚Äî Overall assessment (single-video pivot ‚Äî see ¬ß17)

> Pre-pivot this step picked a winner between Version A and Version B by reading two parallel event/insight lists. With single-video, the task is "summarise THIS demo's brain arc." The synthesizer compiles 8 absolute metrics, then issues a Gemini call that emits an `OverallAssessment` JSON.

Compute **8 absolute metrics** scored against this demo's own timeline:

| Metric | Value | Interpretation |
|---|---|---|
| `mean_appeal_index` | -0.12 | appeal arc skewed negative |

| `pct_reward_dominance` | 18% | 18% of the demo had reward dominant |
| `pct_friction_dominance` | 23% | 23% of the demo had friction dominant |
| `friction_spike_count` | 4 | 4 friction spike(s) detected |
| `motor_readiness_peak` | +0.72 | peak click-readiness over the demo |
| `flow_state_windows` | 0 | 0 flow-state window(s) |
| `bounce_risk_windows` | 1 | 1 bounce-risk window(s) |

Then a final LLM call reads these aggregates + the per-event insights and writes the assessment JSON:

> ```json

>   "summary_paragraph": "The demo opens with strong appeal but slides into sustained cognitive load by t=12s. Friction spiked four times ‚Äî most consequentially at t=18s when the signup form revealed 9 required fields after the inviting CTA. **That's the judging mechanism end to end.** Numbers ‚Üí events ‚Üí insights ‚Üí aggregates ‚Üí overall assessment, each step grounded in the previous.

#### Honest limitations

- Predicts an averaged human brain response. Useful, not ground truth.
- n=1 ‚Äî same demo run twice gives slightly different brain timelines because of agent path variance. For real production use, run each version 3‚Äì5 times and average. For hackathon: single run each, pre-cached.
- Insight quality is bounded by event extraction quality. Bad threshold tuning ‚Üí noisy insights. The Phase 0 spike's smoke-test outputs should be eyeballed for plausibility before locking the thresholds.

### 4.7 MP4 Upload component (Step 2 skip path entry)

**Job:** Two drag-and-drop file zones in Step 2 that accept user-uploaded MP4s, validate them, preview them, and hand them to the same TRIBE pipeline as captured MP4s.

**Accepted formats:**
- MP4 container (H.264 video codec)
- 30s maximum duration (cap matches D7 capture spec)
- 1080p maximum resolution
- ‚â§50MB file size (stops malicious uploads, well above 30s 1080p MP4 size)

**Validation pipeline (client-side first, server-side authoritative):**

```python
# server-side validation in /api/analyze
import ffmpeg

def validate_upload(file_path: str) -> ValidationResult:
    try:
        probe = ffmpeg.probe(file_path)
    except ffmpeg.Error:
        return invalid("Could not parse file. Is it a valid MP4?")

    video_streams = [s for s in probe['streams'] if s['codec_type'] == 'video']
    if not video_streams:
        return invalid("No video track found.")

    v = video_streams[0]
    if v['codec_name'] != 'h264':
        return invalid(f"Codec {v['codec_name']} not supported. Re-encode as H.264.")

    duration = float(probe['format']['duration'])
    if duration > 30.0:
        return invalid(f"Video is {duration:.1f}s. Maximum is 30s.")

    if int(v['width']) > 1920 or int(v['height']) > 1080:
        return invalid(f"Resolution {v['width']}√ó{v['height']} exceeds 1080p maximum.")

    return valid(duration=duration, resolution=(v['width'], v['height']))
```

## 5. TRIBE v2 Service ‚Äî Reference Architecture

> **What this section is:** the proven patterns for wrapping `tribev2` as an HTTP service, captured from a separate prior project (YNeurotrading) that built one for the text pathway. **We are not using their service.** We're building our own from scratch for the video pathway. This section is the eng blueprint we follow ‚Äî architecture choices, file layout, dependency list, env vars, smoke tests. Adapt as needed for our actual deployment.

### 5.1 What this module is

A standalone HTTP service that takes an MP4 screen recording and returns a brain-state timeline. Single output mode for our use case:

- **Timeline** (`/process_video_timeline`) ‚Äî per-TR (1.5s) frame-by-frame stream of brain states (8 UX ROIs from ¬ß5.16), with sliding-window overlay for connectivity-dependent composites.

No agent logic, no persona modulation, no orchestration ‚Äî pure neural interpretation. Aesthesis app sits in front of it and handles everything else.

(Reference patterns below show YNeurotrading's text-pathway endpoints `/process`, `/enqueue`, etc. for context ‚Äî we don't expose those.)

### 5.2 Architecture

Three-process service so the GPU model loads once and HTTP scales independently:

```
Client ‚îÄ‚îÄHTTP‚îÄ‚îÄ> FastAPI (CPU, N uvicorn workers)
                    ‚îÇ
                    ‚ñº  enqueue job
                  Redis (ARQ job queue)
                    ‚îÇ
                    ‚ñº  pop job
                ARQ GPU worker(s) (TRIBE model + whisperx in VRAM)
```

- **Redis** ‚Äî ARQ broker (`redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru`, no persistence)
- **ARQ GPU worker** ‚Äî loads `TribeModel` + masks + weights + signatures + whisperx in `on_startup`; `max_jobs=1` per GPU; registers `process_text` and `process_text_timeline`
- **FastAPI** ‚Äî CPU-only:
  - `GET /health` ‚Üí `{status, gpu_available}`
  - `POST /process` (sync) ‚Äî single aggregated reading, polls every 250ms up to `TRIBE_JOB_TIMEOUT` (default 300s)
  - `POST /enqueue` (async) ‚Äî returns `{job_id}` immediately
  - `GET /job/{job_id}` ‚Äî poll status
  - `POST /process_timeline` (NEW) ‚Äî per-TR timeline (sync), with optional `window_trs` for sliding-window overlay
  - `POST /enqueue_timeline` (NEW) ‚Äî async variant
- Multiple uvicorn + ARQ workers configurable: `API_WORKERS=4`, `GPU_WORKERS=5`, `API_PORT=8000`

### 5.3 The pipeline ‚Äî both modes share Steps 1‚Äì2

| Step | Function | Time | Output |
|---|---|---|---|
| 1 | `run_tribe(text, model)` | 2‚Äì5s GPU | `(n_TRs, 20484)` predicted cortical activation. Pads short text (<30 words) with neutral preamble. Validates non-NaN & shape. Trims last 2 TRs (transformer edge artifact). |
| 2 | `extract_all(preds, masks, weight_maps, signatures)` | <10ms | Dict of 6 ROI timeseries, each shape `(n_TRs,)`. Applies all 7 text-pathway corrections (¬ß5.6). |

#### Path A ‚Äî Single reading (existing)

| Step | Function | Time |
|---|---|---|
| 3 | `extract_stats(ts)` per ROI | <10ms ‚Äî 11 features √ó 6 ROIs |
| 4 | `compute_connectivity(roi_ts)` | <5ms ‚Äî 7 Pearson pairs |
| 5 | `compute_composites(stats, conn)` | <1ms ‚Äî 8 composites |
| 6 | `format_output(...)` | <1ms ‚Äî ~30-line plain-language string |

#### Path B ‚Äî Timeline (NEW)

Two sub-passes that run on `roi_ts`:

**B1. Per-TR pass (one frame per 1.5s)**

For each TR `t = 0..n-1`:

```python
frame = {
    "t_s": t * 1.5,                                  # absolute time in seconds
    "values": {roi: ts[t] for roi, ts in roi_ts.items()},
    "deltas": {roi: ts[t] - ts[t-1] if t > 0 else 0.0 for ...},
    "dominant": max(values, key=values.get),         # which ROI is loudest now
    "dominant_shift": dominant_t != dominant_{t-1},  # bool ‚Äî handover moment
    "local_peak": {roi: ts[t] > ts[t-1] and ts[t] > ts[t+1] for ...},
    "spikes": {roi: deltas[roi] > k * sigma_delta_roi for ...},   # k=1.5 default
    "co_movement": {
        pair: sign(deltas[a]) == sign(deltas[b])     # both rising or both falling
        for pair, (a, b) in PAIRS.items()
    },
    # Per-TR composites (linear formulas collapsed to single-TR)
    "valence":        per_tr_valence(values),
    "arousal":        per_tr_arousal(values),
    "dominance":      per_tr_dominance(values),
    "approach_avoid": per_tr_approach_avoid(values),
    # Per-TR proxies for window-dependent composites
    "regulation_proxy": +1 if deltas["fear_salience"] < 0 and deltas["deliberation"] >= 0
                        else -1 if deltas["fear_salience"] > 0 and deltas["deliberation"] < 0
                        else 0,
    "herding_proxy":   bool(deltas["social_default"] > 0 and deltas["fear_salience"] > 0),
}
```

**Not computed per-TR** (mathematically empty at n=1): all 11 step-3 stats (peak, mean, auc, onset_tr, time_to_peak, rise_time, rise_slope, fwhm, sustained, trajectory, cv, decay_slope), Pearson connectivity (degenerate at n=2), `reactivity` composite, `confidence` composite.

These are emitted by the windowed sub-pass.

**B2. Sliding-window pass**

Defaults: `window_trs=4` (6 seconds), `step_trs=1` (one window per TR).

```python
for start in range(0, n - window_trs + 1, step_trs):
    window_ts = {roi: ts[start:start+window_trs] for roi, ts in roi_ts.items()}
    window_reading = {
        "t_start_s": start * 1.5,
        "t_end_s":   (start + window_trs) * 1.5,
        "stats":         {roi: extract_stats(ts) for roi, ts in window_ts.items()},
        "connectivity":  compute_connectivity(window_ts),
        "composites":    compute_composites(stats, connectivity),
    }
```

The 4-TR minimum matches the existing `len(roi_ts) > 3` guard. No code changes needed in `extract_stats`, `compute_connectivity`, or `compute_composites` ‚Äî they already operate on a 1-D timeseries.

`onset_tr` and `time_to_peak` returned per window are within-window indices. The formatter must convert to absolute time: `absolute_s = (window_start + onset_tr) * 1.5`.

**B3. Timeline formatter**

JSON (machine-readable, default):
```json
{
  "n_trs": 20,
  "tr_duration_s": 1.5,
  "frames": [ {per-TR frame}, ... ],
  "windows": [ {window reading}, ... ],
  "window_config": {"window_trs": 4, "step_trs": 1}
}
```

Plain text (LLM-readable, `format=text`):
```
[Neural state timeline ‚Äî 20 frames @ 1.5s, window=6s]

t=0.0s   fear=0.1 (‚Äî)   reward=0.3 (‚Äî)   delib=0.2 (‚Äî)   social=0.4 (‚Äî)   attn=0.2 (‚Äî)   motor=0.0 (‚Äî)
         dominant: social   v=+0.05  a=0.2  d=+0.1  ap=+0.2

t=1.5s   fear=0.4 (Œî+0.3 ‚Üë)   reward=0.5 (Œî+0.2 ‚Üë)   delib=0.3 (Œî+0.1 ‚Üë)   ...
         dominant: reward   v=+0.10  a=0.5  d=+0.0  ap=+0.3
         co-movement: fear‚Üë+reward‚Üë   spike: reward

[Window 0.0‚Äì6.0s] connectivity: fear‚Üîsocial r=0.42  fear‚Üîdelib r=-0.31  ...
                  composites: reactivity=+2 TRs  regulation=0.0  herding=0.18  confidence=1.05
```

Each line ‚â§120 chars. Blank line between TRs. Windowed reads as a separate trailing block, OR interleaved at `t_start` if `interleave=true`.

### 5.5 Resource loading (one-time per worker)

`init_resources.load_resources()` produces a `Resources` dataclass. Steps:

1. Read `HF_TOKEN` (required).
2. Enable CUDA: `torch.backends.cuda.matmul.allow_tf32 = True`, `cudnn.allow_tf32 = True`, `cudnn.benchmark = True`.
3. `TribeModel.from_pretrained("facebook/tribev2", cache_folder=DATA_DIR/"cache")`.
4. Wrap `model.forward` with `torch.amp.autocast("cuda", dtype=torch.float16)`.
5. Build Schaefer 400 masks: fetch volumetric atlas + fsaverage5; project via `vol_to_surf`; substring-match labels.
6. Load `neurosynth_weights.npz` ‚Üí dict of weight maps.
7. Load `vifs_surface.npy`, `pines_surface.npy` (skip if `vifs_validation_failed` marker exists).
8. Load whisperx (large-v3 ASR + WAV2VEC2_ASR_LARGE_LV60K_960H aligner) and monkey-patch `tribev2.eventstransforms.ExtractWordsFromAudio._get_transcript_from_audio` so TRIBE doesn't reload whisperx per call.
9. Monkey-patch `tribev2.demo_utils.TextToEvents.get_events` to use `edge-tts` (en-US-AriaNeural) instead of gTTS.

### 5.15 Video input pathway (V-JEPA 2 direct) ‚Äî implementation spec

> The YNeurotrading reference exercises only the text+audio path. The V-JEPA 2 video path is fully supported by the underlying `tribev2` package, just not surfaced in their service. **We build a fresh service that uses only the video pathway**, exposing one endpoint: `/process_video_timeline`. This section is the implementation spec.

#### 5.15.1 TRIBE v2 video API (verbatim from `facebookresearch/tribev2` demo)

```python
from tribev2.demo_utils import TribeModel, download_file
from pathlib import Path

CACHE_FOLDER = Path("./cache")
model = TribeModel.from_pretrained("facebook/tribev2", cache_folder=CACHE_FOLDER)

preds, segments = model.predict(events=df)
# preds.shape == (n_timesteps, n_vertices)
```

`get_events_dataframe` is overloaded ‚Äî pass exactly one of `text_path=`, `audio_path=`, or `video_path=`.

#### 5.15.2 Encoder details

- **V-JEPA 2-Giant:** frozen feature extractor. 64 frames covering preceding 4s of video per time bin. Output frequency 2 Hz (one embedding per 500ms). Embedding dim `D_video = 1,280`.
- **DINOv2:** also extracted alongside V-JEPA 2 (visual features, complementary).
- **Spatial preprocessing:** patch tokens are spatially averaged before fusion. The TRIBE paper explicitly flags: *"expected to deteriorate encoding performance in low-level visual areas."* For UI demos this means cursor/text-position/focus signals living in retinotopic V1/V2/V4 are blurred. **Not a problem for our 6 ROIs** ‚Äî they're high-level integration regions (salience network, limbic, control, default mode, somatomotor, dorsal attention), not retinotopic. See premise 3 in ¬ß6.
- **Multimodal fusion:** all three streams aligned to a uniform 2 Hz grid ‚Üí layer-wise concatenated ‚Üí projected through shared linear layer to `D = 1,024` ‚Üí concatenated again ‚Üí temporal Transformer ‚Üí cortical voxel head.
- **Cortical projection:** full TRIBE v2 outputs ~70k voxels; the `facebook/tribev2` HF release / `demo_utils` path used by YNeurotrading downsamples / re-projects to **fsaverage5 (20,484 vertices)** ‚Äî same shape as the text path, drop-in compatible with everything downstream of step 1.
- **fMRI alignment:** 2 Hz feature stream collapsed to TR rate (TR = 1.49s ‚âà 0.67 Hz) via adaptive average pooling.
- **Hemodynamic lag offset:** **predictions are offset by 5s in the past** to compensate for BOLD delay. Important when aligning brain frames to actual video timestamps for UI overlays ‚Äî see open question ¬ß10.12.

### 5.16 UX-tuned ROI keyset (canonical and only)

> **Status: canonical.** The only keyset our service exposes. No legacy keyset, no `KEYSET` flag ‚Äî this is what `extract_all` produces. The YNeurotrading reference used a different, generic emotional/cognitive keyset (fear/reward/delib/social/motor/attn) tuned for narrative text. We don't carry it forward.
>
> **Known implementation gaps to resolve when building** (none are design blockers):
> 1. The `NETWORK_KEYS_UX` dict (¬ß5.16.2) simplifies three mappings vs. the table anatomy ‚Äî `trust_affinity`, `surprise_novelty`, `aesthetic_appeal` won't fully match the published signature without Schaefer-parcel-subset post-processing or signature subtraction. v1 yields a directionally correct, magnitude-attenuated signal for these three.
> 2. `_Vis_` Yeo network must be included in our Schaefer mask build (it wasn't in the YNeurotrading reference's 6-network build). Required for `visual_fluency`.
> 3. `visual_fluency`'s `_Vis_ ‚àí 0.5 √ó _DorsAttn_` two-pass extraction needs a post-step in `extract_all` (the dict can't express it).
> 4. ¬ß5.17 Subcortical extension referenced in ¬ß5.16.7 ‚Äî TBD / future scope.
> 5. `visual_fluency` is partially retinotopic (`_Vis_` ‚âà V1/V2/V4) ‚Äî degraded by V-JEPA 2 spatial averaging. Other 7 keys unaffected. See premise 3 in ¬ß6.

This section defines the keyset, formulas, composites, and connectivity pairs.

#### 5.16.1 Why re-key

- **Current keys are too general.** `social_default` is the entire DMN, but for UX the load-bearing DMN signal is *aesthetic appreciation* (Vessel et al. 2019) ‚Äî and most UI screens have no social content, so the broad "social" frame is misleading. `reward_limbic` mixes hedonic "liking" with anticipatory "wanting" (Berridge), but for conversion intent we want NAcc anticipation specifically. `deliberation` is Yeo Control network, but the cognitive-load literature points more sharply at DLPFC + dACC.
- **The video pathway re-enables signatures that text-pathway disabled.** ¬ß5.15.5 already removes corrections #1 (VIFS), #2/#3 (motor demean), #6 (text padding). VIFS becomes a real fear/threat readout. Wager-lab-style multivariate signatures (NPS-style for various states) become viable as composite features.
- **TRIBE v2 outputs subcortex too.** The Meta release predicts both ~20k cortical fsaverage5 vertices AND ~8.8k subcortical voxels (`run_subcortical` grid in the GitHub repo; documented in Meta's blog and MarkTechPost coverage). The current YNeurotrading deployment only mounts the cortical surface via Schaefer 400. Three of the most decision-relevant regions for UX ‚Äî **nucleus accumbens** (Knutson SHOP task), **amygdala** (trust/threat), **anterior hippocampus** (memorability, prediction error) ‚Äî are subcortical. Exposing them needs a service-level extension (¬ß5.17 below). Without it, we fall back to cortical proxies for those keys.

#### 5.16.3 Per-TR composites ‚Äî UX judgments as formulas

Each composite derives from a published mechanism. All formulas operate on the per-TR `values` dict.

```python
# Vessel 2019: aesthetic appeal = DMN engagement co-occurring with hedonic positive
# Penalize cognitive load (workload kills the aesthetic experience) and friction
def appeal_index(v):
    return (0.45 * v["aesthetic_appeal"]
          + 0.25 * v["reward_anticipation"]
          - 0.20 * v["cognitive_load"]
          - 0.10 * v["friction_anxiety"])

# Knutson 2007 SHOP task formula for purchase prediction:
#   product preference (NAcc) ‚àí price aversion (insula) ‚Üí purchase
# We modulate by motor_readiness because the user has to be in a clickable state.
def conversion_intent(v):
    nacc = v["reward_anticipation"]
    insula = v["friction_anxiety"]
    motor = v["motor_readiness"]
    drive = (nacc - insula) / (nacc + insula + 1e-6)   # [-1, +1]
    return motor * max(drive, 0.0)                      # only when motivation is positive

# Reber 2004 processing fluency: ease of parsing - effort to parse
def fluency_score(v):
    return v["visual_fluency"] - 0.5 * v["cognitive_load"]

# Mende-Siedlecki / FeldmanHall: trust = vmPFC integration ‚àí amygdala threat
def trust_index(v):
    return v["trust_affinity"] - 0.6 * v["friction_anxiety"]

# Yerkes-Dodson inverted-U: engagement peaks at moderate load
def engagement_depth(v):
    load = v["cognitive_load"]
    optimal = max(0.0, 1.0 - 2.0 * abs(load - 0.5))    # peaks at load=0.5
    return v["aesthetic_appeal"] * optimal

# Sign of surprise (delight vs shock) determined by reward/fear balance.
# Hippocampal/dACC PE magnitude √ó direction.
def surprise_polarity(v):
    direction = v["reward_anticipation"] - v["friction_anxiety"]
    sign = 1.0 if direction >= 0 else -1.0
    return sign * v["surprise_novelty"]

# Bainbridge 2017: memorability ~ anterior hippocampus + parahippocampal late response.
# Cortical proxy: aesthetic_appeal √ó surprise_novelty (memorable = beautiful + unexpected).
def memorability_proxy(v):
    return v["aesthetic_appeal"] * v["surprise_novelty"]

# UX-specific dominance: "is the user being driven by appeal or by friction?"
# Replaces the generic per_tr_dominance.
def ux_dominance(v):
    pos = v["aesthetic_appeal"] + v["reward_anticipation"]
    neg = v["cognitive_load"]   + v["friction_anxiety"]
    return (pos - neg) / (pos + neg + 1e-6)
```

#### 5.16.4 Window composites ‚Äî patterns over 6s

These need ‚â•4 TRs (6s) and are emitted by the windowed sub-pass (B2 in ¬ß5.3).

| Composite | Formula sketch | UX meaning |
|---|---|---|
| `flow_state` | mean(`engagement_depth`) > 0.6 AND std(`appeal_index`) < 0.2 over the window | sustained absorbed engagement |
| `decision_clarity` | `motor_readiness` rising AND `cognitive_load` falling within the window (Pearson r < ‚àí0.3) | "I figured it out, now I act" ‚Äî clean conversion path |
| `bounce_risk` | `cognitive_load` AND `friction_anxiety` both > 1œÉ above their means, with `motor_readiness` flat | confused user about to leave |
| `hook_strength` | `appeal_index` peak in first 3 TRs of run (Lindgaard 2006: 50ms first-impression window, BOLD-shifted) | how strong the page's first impression is |
| `aesthetic_dwell` | `aesthetic_appeal` sustained > 0.5œÉ for ‚â•3 consecutive TRs | the user is letting the page be beautiful at them |
| `friction_burst` | `surprise_polarity` < ‚àí0.5 with `friction_anxiety` spike co-occurring | bait-and-switch / dark pattern moment |

## 7. Approaches Considered

### Approach A ‚Äî "Brain Replay" (story-driven side-by-side)

Two browser agents demo their URL in parallel. Live 10fps frame streams render side-by-side on the frontend during the run. Each MP4 POSTs directly to TRIBE v2's `/process_video_timeline` endpoint. An LLM synthesizer reads both brain timelines + screenshots + agent action logs and produces 3‚Äì5 timestamped insights per version + a head-to-head verdict. UI is the same side-by-side panel ‚Äî live frames during run, MP4 + ROI charts + insights + verdict after.

- **Effort:** M-L (2‚Äì3 days for solid hackathon demo)
- **Risk:** Medium
- **Reuses:** existing TRIBE deployment + new video endpoint, BrowserUse / Playwright, Recharts
- **Demo magic:** HIGH ‚Äî live agent streams + cortex visualizations + side-by-side comparison + AI verdict
- **Pros:**
  - True "two versions of anything" pitch
  - Live frame streams during run = no dead UI ("watching the AI demo your product")
  - Visually impressive (split screen + brain charts)
  - Judges can interact (click an insight, see the moment in the demo)
  - Pipeline is decoupled ‚Äî each component can fail/swap independently
- **Cons:**
  - More moving parts than B
  - Two parallel demos = 2√ó failure surface during live presentation

### Approach B ‚Äî "Brain Diff" (single-flow tight comparison)

User specifies ONE specific flow ("complete signup," "find pricing," "buy a t-shirt"). Same scripted flow runs on both URLs. Brain timelines compared for that one flow. Output is a focused diff: "Version A's signup costs the user 30% more cognitive load," with 3‚Äì5 specific recommendations.

- **Effort:** S-M (1‚Äì2 days)
- **Risk:** Low
- **Reuses:** Playwright (scripted), TRIBE deployment, LLM
- **Demo magic:** MEDIUM ‚Äî tighter and more credible, less visually striking
- **Pros:**
  - Easier to ship and demo reliably
  - Tighter comparison story (apples to apples)
  - Recommendations more concrete (specific to one task)
- **Cons:**
  - Loses the "two versions of anything" pitch ‚Äî narrows to "two versions of this one flow"
  - Less of a "whoa" moment
  - Each new flow requires a new script

### Approach C ‚Äî "Continuous Brain Monitor" (live agent + live brain feed)

Agent runs the demo in real time on stage. Brain feed streams alongside the video like a Twitch overlay. Insights appear as the demo plays. End with the verdict.

- **Effort:** L-XL (3‚Äì5 days, requires streaming architecture)
- **Risk:** High
- **Reuses:** TRIBE service (would need streaming add-on or chunked posting), live frontend
- **Demo magic:** VERY HIGH if it works ‚Äî unforgettable
- **Pros:**
  - Most cinematic demo possible
  - "The brain is reacting LIVE on stage" is a story unto itself
- **Cons:**
  - 5s GPU latency per TRIBE call ‚Üí not truly live, has lag
  - Streaming + sync between video and brain is fiddly
  - Highest probability of nothing-works-on-stage
  - Feels gimmicky if results are jittery

## 8. Recommended Approach

**Approach A ‚Äî Brain Replay.**

**Why:**
- Best balance of "whoa" and "works." Live agent streams + side-by-side + brain overlay + AI verdict is genuinely novel and looks like a product, not a science fair poster.
- Decoupled architecture: each component (browser agent, live streamer, TRIBE call, synthesizer, UI) can be built and tested independently. Parallel work is possible.
- Recovers gracefully from partial failure: if BrowserUse breaks on one URL, swap to a pre-recorded MP4. If TRIBE is slow, show cached results. If the synthesizer is shaky, have a hand-tuned example as the demo flow.

**Bridge:** Direct video input via V-JEPA 2 (¬ß4.3 + ¬ß5.15). Single path, no fallback designed ‚Äî if the Phase 0 spike (¬ß5.15.6) fails, escalate per ¬ß5.15.7.

**Why not Approach C ("Continuous Brain Monitor"):** the 5s GPU latency per TRIBE call + streaming sync makes "live brain" inherently lagged. The demo magic depends on a smooth real-time effect that we can't actually deliver. Better to show a finished side-by-side replay than a janky live brain overlay. (Note: the live frame stream of the browser agent ‚Äî ¬ß4.2b ‚Äî IS truly live and low-lag because it doesn't go through TRIBE.)

**Why not Approach B ("Brain Diff"):** narrowing to one flow loses the "two versions of anything" pitch ‚Äî that's the headline. Brain Replay keeps it.

## 13. Glossary

- **TR** ‚Äî repetition time. The fundamental time unit in fMRI; here, 1.5 seconds. TRIBE outputs one brain-state vector per TR.
- **fsaverage5** ‚Äî a low-resolution standard cortical surface mesh (~20,484 vertices total, ~10k per hemisphere) used by FreeSurfer/Neurosynth/TRIBE for cross-subject comparison.
- **ROI** ‚Äî region of interest. We aggregate the 20k cortical vertices into 6 named functional regions (see ¬ß5.4).
- **VLM** ‚Äî vision-language model. Multimodal LLM that can see images.
- **Composite** ‚Äî derived metric combining multiple ROI activations (e.g., `valence` = weighted sum of reward minus fear).
- **VIFS / PINES** ‚Äî established multivariate brain signatures (fear, negative affect). Disabled in text-pathway corrections (¬ß5.6).
- **NiMARE** ‚Äî Python library for neuroimaging meta-analysis. Used to build the Neurosynth weight maps.
- **ALE** ‚Äî Activation Likelihood Estimation, a meta-analysis method NiMARE uses to convert thousands of fMRI papers into a brain weight map per term.

### 2026-04-25 ‚Äî TRIBE inference profiling + audio-patch reversal + V-JEPA bake

After the first end-to-end live run measured ~150 s for a 13.74 s MP4 (an A100-40GB at $1.30/hr should not be running 11√ó slower than the source), profiled the call stack against upstream `neuralset==0.1.0` source (extracted from PyPI), applied the cheap fixes that don't reach into tribev2, and established a hard policy boundary for the rest.

**New rule: don't touch tribev2 internals.**

Definition: no monkey-patches into `neuralset.*` or `tribev2.*`, no overriding configuration of post-construction model objects, no changes that reach inside `TribeModel`. Acceptable: pre-/post-processing at the request boundary, Modal infra config, and any Python code we own. Reasoning: tribev2/neuralset is research code at v0.0.x / v0.1.0; upstream method names and signatures will move; every monkey-patch is a hidden coupling we'll silently break on `pip install` updates. The pre-existing audio monkey-patches predated this rule and were grandfathered until reversed in this session.

This rule deletes from the option set everything that previously looked like a "5‚Äì10√ó speedup": batched chunk forwards, decord pre-decode, `torch.compile` of the inner model, BF16 autocast wrapping, `frequency`/`num_frames`/`max_imsize` tuning. Realistic floor under the rule: ~150 s on A100-40GB, ~60‚Äì90 s on H100, for a 13 s clip. Going below ~60 s requires either touching tribev2 or replacing it.

**Decisions applied in this session:**

|---|---|---|
| A | Pre-fetch `facebook/vjepa2-vitg-fpc64-256` (~5 GB) into the Modal image's HF cache via `huggingface_hub.snapshot_download` at build time. ## 17. Pivot: A/B comparison ‚Üí single-video analysis

**Date:** 2026-04-25.
**Pitch line shift:** "We don't ask users which version they like. We watch their brains." ‚Üí **"Demo anything. See the analysis."**

### Why
The A/B comparison shape was forcing every Aesthesis user into a "choose between two demos" funnel. That excludes the most common first-use scenario: someone who has *one* product and wants to know how the brain reads it. The pivot opens the surface area to landing pages, signup flows, dashboards, mobile-app captures ‚Äî anything you can record.

### What changed in the contract
The whole `VersionTag = "A" | "B"` discriminator is gone. Every request, every record, every component is now single-subject.

| Concept | Pre-pivot | Post-pivot |
|---|---|---|
| `POST /api/analyze` body | `video_a` + `video_b` | `video` |
| `Event.version`, `Insight.version` | `"A" | "B"` | (removed) |
| `AggregateMetric` | `{name, a, b, edge, edge_description}` | `{name, value, interpretation}` |
| `Verdict` (winner picker) | exists | replaced by `OverallAssessment` |
| `OverallAssessment` | ‚Äî | new: `{summary_paragraph, top_strengths, top_concerns, decisive_moment}` |
| `AnalyzeResponse` | `{a: VersionResult, b: VersionResult, verdict, ...}` | flattened: `{video_url, duration_s, timeline, events, insights, aggregate_metrics, overall_assessment, ...}` |
| Synthesizer Gemini calls | 3 (insights √ó 2 + verdict) | 2 (insights + assessment) |
| Wall time (Step 2) | 15-25s | 6-13s |

### What stayed the same
- TRIBE service (`¬ß5.x`) ‚Äî the GPU API was always per-video. `/process_video_timeline` is unchanged.
- Schaefer masks, Neurosynth weight maps, ROI keyset (`NETWORK_KEYS_UX`) ‚Äî all per-vertex math is unchanged.
- Event extraction (`events.py`) ‚Äî same 7 event types, same caps, just no `version` arg.
- `aggregate_metrics` block keeps all 8 metric names; only the per-record shape changed (now absolute, not comparative).

### Frontend layout (replaces ¬ß4.6)
Single-column results page:
1. **Top:** `VideoPlayer` (single MP4) + 3D brain at the current TR.
2. **Middle:** `BrainChart` ‚Äî one chart with 8 ROI lines, insight regions overlaid.
3. **Bottom-left (2/3 width):** `InsightCard`s, one per timestamped event.
4. **Bottom-right (1/3 width):** `OverallAssessmentPanel` ‚Äî summary paragraph + top_strengths + top_concerns + decisive_moment.

Files renamed: `VerdictPanel.tsx` ‚Üí `OverallAssessmentPanel.tsx`, `VideoPlayerSynced.tsx` ‚Üí `VideoPlayer.tsx` (no second video to sync to).

### Out of scope for this pivot
- Restoring A/B comparison as a separate `/compare` endpoint. Plausible v2 feature; the per-video pipeline survives unchanged so it would just be an orchestration layer on top.
- Per-dimension absolute baselines for metric `interpretation` strings (we use within-video phrasing instead).
- Eval suite for `OverallAssessment` quality ‚Äî this needs a held-out set of demos with ground-truth UX feedback. Separate effort.

### Audio pathway is removed end-to-end (2026-04-25)
Aesthesis is video-only ‚Äî **no audio anywhere in the pipeline**, ever. The user's intent: "delete all audio related operations, only video and images should remain." A single, request-boundary mechanism enforces this without touching tribev2 internals:

**API-boundary strip (load-bearing).** ``tribe_neural/api._strip_audio_track`` rewrites the incoming MP4 with ``ffmpeg -an -c:v copy`` (sub-second, no re-encode) before the path is forwarded to TRIBE. The file that reaches the pipeline has zero audio streams. Failure of the strip raises ``PipelineError`` and returns 500 ‚Äî silently passing audio to TRIBE would re-introduce the whisperx bottleneck and is now explicitly rejected.

What an audio-stripped MP4 does to TRIBE's pipeline:
- ``ExtractAudioFromVideo`` reads ``video.audio`` via moviepy ‚Üí ``None`` ‚Üí produces zero Audio events.
- ``tribev2.main`` then prunes the audio and text extractors with "Removing extractor ‚Ä¶ as there are no corresponding events".
- ``ExtractWordsFromAudio`` (whisperx, the original 2:30 / 30 s bottleneck) and the Wav2Vec-BERT raw-waveform encoder are never reached.

What this changes in TRIBE's multimodal forward pass:
- **Video frames ‚Üí V-JEPA 2** ‚Äî unchanged, the load-bearing modality.
- **Raw audio ‚Üí Wav2Vec-BERT** ‚Äî never instantiated for this request (extractor pruned).
- **Transcript words ‚Üí LLaMA 3.2 text tokens** ‚Äî never instantiated for this request (extractor pruned).

**History.** Earlier revisions used a defense-in-depth design with an additional ``TribeRunner._patch_audio_pipeline_to_empty`` monkey-patch on ``ExtractAudioFromVideo._run`` and ``ExtractWordsFromAudio._get_transcript_from_audio``. That patch was removed once we adopted the rule "don't touch tribev2 internals" (see ¬ß15 entry 2026-04-25 ‚Äî "TRIBE inference profiling + audio-patch reversal + V-JEPA bake") ‚Äî the audio strip alone is sufficient because tribev2's own pruning logic ("Removing extractor ‚Ä¶ as there are no corresponding events") handles the empty-audio case natively. Pre-pivot, DESIGN.md ¬ß5.5 step 8 prescribed a whisperx GPU monkey-patch; we went further and removed audio entirely. Our 8 UX-tuned ROIs (``NETWORK_KEYS_UX``) are visual / cognitive / emotional networks ‚Äî auditory cortex is not exposed in the keyset, so removing the audio modality has no impact on the surfaced ROI signals.

Wall-time impact of removing audio: the whisperx 2:30-min CPU bottleneck is gone end-to-end. However, removing audio is necessary but not sufficient ‚Äî the V-JEPA-2-vitg video encoding loop (27 √ó B=1 sequential forwards) is now the dominant cost. Measured wall time for a 13.7 s clip on A100-40GB cold container = **147.9 s** (full breakdown in ¬ß15 entry 2026-04-25). Earlier text-pathway-derived budgets in ¬ß5.13 do not apply to the video pathway.
